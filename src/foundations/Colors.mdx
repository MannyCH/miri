# Color Tokens

Miri uses a semantic color system with tokens organized into categories based on their usage.

All colors are available as CSS custom properties and can be used directly in your styles.

## Usage Examples

### Button Component

```css
.button-primary {
  background: var(--color-fill-brand-strong);
  color: var(--color-text-inverted);
  border: 1px solid var(--color-stroke-brand-strong);
}

.button-primary:hover {
  background: var(--color-fill-hover);
}

.button-primary:disabled {
  background: var(--color-fill-disabled);
  color: var(--color-text-disabled);
  border-color: var(--color-stroke-disabled);
}
```

### Card Component

```css
.card {
  background: var(--color-background-raised);
  border: 1px solid var(--color-stroke-weak);
}

.card-title {
  color: var(--color-text-strong);
}

.card-description {
  color: var(--color-text-weak);
}
```

### Input Field

```css
.input {
  background: var(--color-background-sunken);
  border: 1px solid var(--color-stroke-weak);
  color: var(--color-text-strong);
}

.input:focus {
  border-color: var(--color-stroke-focus);
}

.input[aria-invalid="true"] {
  border-color: var(--color-stroke-error-strong);
  background: var(--color-fill-error-weak);
}
```

## Best Practices

✅ **DO:** Use semantic tokens (`--color-text-strong`) instead of primitive colors  
✅ **DO:** Test color contrast for accessibility (WCAG AA minimum)  
✅ **DO:** Use state-specific tokens for hover/focus/disabled states  

❌ **DON'T:** Hardcode color values  
❌ **DON'T:** Use primitive colors directly in components  
❌ **DON'T:** Override semantic tokens with custom values
